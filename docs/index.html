Install

Three ways. Pick one.

Claude Code loads skills from ~/.claude/skills/<name>/. Pick any of three.

1. Quickest: git clone into the skills dir

Clone directly into Claude Code's skills directory. Restart your session and /prototype is loaded.

git clone https://github.com/kurenn/claude-prototype ~/.claude/skills/prototype

2. Recommended for contributors: ./install.sh

Clone somewhere you will hack on it, then run the install script. Symlink default means edits to your clone flow through to future /prototype invocations with no reinstall.

git clone https://github.com/kurenn/claude-prototype ~/code/claude-prototype
cd ~/code/claude-prototype
./install.sh            # symlinks, live edits
./install.sh copy       # or: independent copy

3. Manual one-liner

Already have a clone somewhere. Symlink it.

ln -s /absolute/path/to/claude-prototype ~/.claude/skills/prototype

Verify

Restart your Claude Code session, then type /prototype or describe a prototype idea. The skill triggers automatically on words like prototype, mockup, demo, pitch page, click-through.

Reinstall commands

You already have /prototype installed. If you want to reinstall or update, the three options are:

cd ~/.claude/skills/prototype && git pull     # if installed via clone
cd ~/code/claude-prototype && git pull       # if installed via ./install.sh
rm ~/.claude/skills/prototype                  # uninstall

Features

What you get.

Most AI-generated prototypes look like every other AI-generated prototype. Purple gradients, gradient text, nested cards, lorem ipsum, dead buttons, no lifecycle states. This skill bakes in the opposite.

Discovery Q&Abefore building

Six questions: tone, inspiration, audience, scope, content. Prevents generic "modern SaaS" output.

prompt-refiner turns the answers into a tight build spec. Auto-installed.

Design shapingvia impeccable

impeccable teach + shape produces a per-prototype DESIGN.md with tokens. Palette, typography, spacing, motion.

After build, audit + detect + critique verify. 25 anti-pattern rules, Nielsen heuristics, UX scoring. Auto-installed.

Always-visible control barthree switchers, two actions

Theme. 3 per-prototype themes. Design tokens flip at runtime. URL-shareable.

Layout. 2 to 4 per-prototype variants. Grid / Gallery / List, or density, or grid-N. Chosen per product.

Persona. 2 to 4 lifecycle states (new user / active / power, or empty / busy). Content swaps via js/data.js.

🔗 Share. Copies a URL reproducing the exact screen: theme, layout, persona, modal, tab.

💬 Feedback. Always-on pin-to-element commenting. Exports JSON for /prototype apply-feedback.

Interaction statesthe layer demos die without

Loading spinners, toasts, empty states, 404 page. Declarative via data-loading, data-toast, data-confirm attributes.

Skeleton loaders on filter changes and first load.

URL stateevery screen shareable

Every interactive screen state lives in the URL. Share the URL, reproduce the screen. Opens directly to a modal, a filter result, a tab.

Apply-feedback workflowJSON in, fixes out

Reviewers pin comments. You run /prototype apply-feedback <file>. The skill applies each fix in place, re-runs assessment, archives the JSON.

VariantsA/B your pitch

/prototype variant "more corporate" forks into variants/<slug>/. Original untouched. Shape, palette, type re-applied for the new vibe.

No-cache dev serverships with every prototype

serve.py in every generated folder. Prevents the "my edits aren't showing" browser cache trap.

Zero build stepTailwind CDN, vanilla JS

No webpack, no Vite, no Node. Tailwind via CDN. A handful of small JS files. Runs anywhere a browser opens.

Output

What a generated prototype looks like.

One folder in your working directory. Every file readable, every file editable, no build artifact.

acme-demo/
├── index.html
├── <screen>.html
├── 404.html
├── css/
│   ├── styles.css       # theme tokens + control bar + skeleton + empty states
│   └── feedback.css
├── js/
│   ├── state.js         # URL state + share + history drawer (Shift+?)
│   ├── theme.js         # data-theme switcher
│   ├── layout.js        # data-layout switcher
│   ├── data.js          # personas + shared content
│   ├── persona.js       # data-persona switcher
│   ├── ui.js            # loading / toast / skeleton / confirm helpers
│   ├── app.js           # page interactions (modals, tabs, filter chips, composer)
│   └── feedback.js      # pin-to-element overlay (always on)
├── assets/images/
├── serve.py             # no-cache dev server
├── DESIGN.md            # tokens + rationale
├── DEMO.md              # presenter click-through with persona map
└── README.md            # how to run, what's fake, iteration commands

Run it:

cd acme-demo
python3 serve.py
# open http://localhost:8000

Then click through the control bar at the bottom to toggle theme, layout, and persona. Click 🔗 to copy a URL reproducing the current screen. Click 💬 to pin feedback on any element.

Subcommands

/prototype variant "<vibe>"

Fork the current prototype into variants/<slug>/. Re-shapes tokens, typography, palette for the new vibe. Original untouched.

/prototype apply-feedback <file>

Read the pinned-comment JSON, apply each fix in place, re-run assessment, archive under feedback/applied/.

Companion skills

Auto-installed on first use.

/prototype runs a preflight step on every invocation that ensures these companion skills are available. Missing ones are installed automatically.

impeccable

Design direction and 3-pass assessment: audit, detect, critique. 23 subcommands. Strongly recommended.

npx -y skills add pbakaus/impeccable --global --yes

prompt-refiner

Turns Q&A answers into a structured build spec.

git clone https://github.com/kurenn/prompt-refiner-skill ~/.claude/skills/prompt-refiner

claude-in-chrome (MCP)

Screenshot grid across themes and breakpoints. Console error sweep. Configured in Claude Code MCP settings, not auto-installed.

If auto-install fails (no npx, no network), /prototype falls back to built-in checks and notes it in the final report.

Run the preflight manually anytime:

bash ~/.claude/skills/prototype/ensure-deps.sh          # prompts before installing
bash ~/.claude/skills/prototype/ensure-deps.sh --yes    # non-interactive
bash ~/.claude/skills/prototype/ensure-deps.sh --check  # report status only

Principles

What the skill enforces.

Rules, not vibes. Every principle corresponds to a check in the build loop.

  1. Ask before building. Discovery Q&A is load-bearing. Prevents generic output.
  2. No lorem ipsum, ever. Realistic content matched to the product domain.
  3. Every button goes somewhere. Dead buttons in a sales demo kill the pitch.
  4. Always interactive. Loading states, toasts, empty states, 404. Skeleton loaders on data-changing interactions.
  5. Control bar never wraps. flex-wrap: nowrap is load-bearing. A two-line bar breaks reviewer trust.
  6. Responsive by default. Verified at 375, 768, 1440.
  7. No build step. Tailwind CDN only. If a prototype needs webpack, it's not a prototype.
  8. Theme-safe colors. Every color reference is a CSS var. Flipping themes never breaks layout.
  9. Persona-aware content. Names, counts, lists driven from js/data.js. Not hardcoded in HTML.
  10. URL state is shareable. Share a URL, reproduce the exact screen: theme, layout, persona, modal, tab.
  11. Scope discipline. 4 screens asked, 4 screens shipped.
